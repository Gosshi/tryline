# Codex プロンプトテンプレート

Codex に作業を委譲するためのテンプレート集。Owner が角括弧部分を埋めて使用します。

## テンプレート: 仕様書を実装する

```
`/specs/[仕様書ファイル.md]` の仕様を実装してください。

コンテキスト:
- プロジェクト規約は `CLAUDE.md` を読む
- システム設計は `/docs/architecture.md` を読む
- 過去の判断は `/docs/decisions.md` を読む
- `/lib` と `/app` に既に確立されているパターンに従う

要件:
- 受け入れ条件セクションのすべてを実装する
- 「スコープ対象外」にある項目は実装しない
- 受け入れ条件項目に対するテストを書く
- 曖昧な箇所があれば末尾に質問として列挙する。推測しない

完了時:
- 実装内容、変更ファイルを要約する
- 仕様書からの逸脱があれば理由を明示する
- Owner への未解決の質問があれば記載する
```

## テンプレート: 小さな個別変更

```
タスク: [一文での説明]

影響を受けそうなファイル: [推測]

制約:
- [具体的な制約]

受け入れ条件:
- [具体的なテスト]

`CLAUDE.md` の規約に従う。
```

## テンプレート: バグ修正

```
バグ: [説明]

再現手順:
1. [ステップ]
2. [ステップ]
期待動作: [起きるべきこと]
実際の動作: [実際に起きていること]

調査済み:
- [仮説、または排除済みの原因]

原因と思われる場所: [推測]

成果物: 修正 + リグレッションテスト
```

## テンプレート: リファクタリング

```
リファクタ対象: [ファイル／モジュール]

目的: [なぜ]

スコープ:
- 対象: [変更する内容]
- 対象外: [挙動を含め変更してはならないもの]

制約:
- 既存テストがすべてパスする
- 公開 API のブレーキングチェンジ禁止
- 確認なしに新しい依存を追加しない

成果物: リファクタ済みコード + 必要に応じて新規テスト
```

## Codex プロンプトで避けるべきアンチパターン

- プロダクトの判断を Codex に委ねる（判断は Owner、助言は Claude Code の役割）
- 仕様書への参照を省く（Codex には常に権威ある仕様書が必要）
- 複数の仕様書を 1 つのプロンプトにまとめる（順次実行する）
- 「まず CLAUDE.md を読む」を省略する（Codex はセッション毎にプロジェクトコンテキストが必要）
- 曖昧な受け入れ条件（「もっと良くして」）— 常に具体的でテスト可能に
