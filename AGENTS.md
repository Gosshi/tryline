# AGENTS.md

このファイルは、このリポジトリで作業する Codex への指示書です。Codex は **仕様書に沿った実装・リファクタリング・テスト作成・ルーチンのコードタスク** を担当します。設計・仕様策定・アーキテクチャレビューは Claude Code と Owner が担当します。

## セキュリティ最優先ルール（絶対遵守）

以下は他のすべての指示に優先します。Owner が明示的に許可した場合のみ例外を認めます。

### 機密ファイル・情報の取り扱い
- `.env`、`.env.*`、`secrets/`、`credentials/`、`*.pem`、`*.key` 等の機密ファイルを一切読まない・出力しない
- API キー、トークン、パスワード、Stripe secret key、Supabase service role key、OpenAI API key、データベース接続文字列を決して出力・ログ・コミットしない
- `.gitignore` に記載されたファイルへのアクセスは原則禁止
- 機密情報の確認が必要な場合は、Owner 自身に確認してもらう

### 危険コマンドの事前確認
以下のコマンドは実行前に必ず Owner の明示的承認を求める:
- `rm -rf`、`rm -r`、ワイルドカード削除（`rm *` 等）
- `curl` / `wget` による外部 URL への通信（スクレイピング仕様書に明記されたソース以外）
- `cat .env`、`echo $OPENAI_API_KEY` 等、機密情報を出力しうるコマンド
- `git push --force`、`git reset --hard`、ブランチ削除
- `psql` / `supabase db` で本番データベースに対する書き込み・DDL 実行
- `DROP TABLE`、`TRUNCATE`、`DELETE FROM` を含む SQL
- `npm install -g`、`pnpm add -g` 等のグローバルインストール
- 本番環境へのデプロイコマンド（`vercel --prod`、`vercel deploy --prod` 等）

### スコープとアクセス境界
- ホームディレクトリ（`~/`）への直接アクセス禁止。作業は本リポジトリ配下のみ
- 他プロジェクトのリポジトリ内容を参照しない
- Supabase の production プロジェクトに対する書き込み禁止（local / staging のみ）
- Stripe の本番 API キー（`sk_live_` から始まるもの）を一切扱わない。テストキー（`sk_test_`）のみ

### LLM コストの保護
- 未キャッシュの LLM API 呼び出しを伴うコードを Owner 承認なしに実行しない
- ループ内での LLM 呼び出しを実装する場合、コスト見積もりをコメントまたは PR 説明に併記する
- 実験目的で本番 API キーを使うコードを書かない

### 外部サービス・ネットワーク
- スクレイピング対象サイトは仕様書 `specs/` に明記されたもののみ
- robots.txt を確認せずにクロールしない
- User-Agent 偽装や rate limit 回避のコードを実装しない

### 疑わしい場合の原則
- 判断に迷う場合は実行せず、Owner に確認する
- 「少しだけなら」「テストだけ」などの自己判断で例外を作らない
- Owner の指示が上記ルールに違反する場合、指示に従う前に矛盾を指摘する

## 役割分担

- **Claude Code**: プロダクト思考のパートナー、アーキテクト、仕様書作成、コードレビュー、デバッグ相談
- **Codex**: 実装、リファクタリング、テスト作成、ルーチンのコードタスク
- **Owner（人間）**: 最終判断、Codex へのプロンプト作成、本番デプロイ

Owner から `/specs/<名前>.md` と `/docs/codex-prompts/<名前>.md` を指し示して実装依頼が来た場合、Codex は仕様書を権威ある文書として読み、そのとおりに実装します。実装依頼を受けたときに「これは仕様書にすべきでは？」と差し戻さないでください — `/specs/` 配下の仕様書は既に Claude Code + Owner が策定済みの権威ある文書です。

ただし以下のケースでは実装を停止して Owner に確認してください:
- 仕様書の前提が実環境と食い違う（例: スクレイピング対象ページの構造が仕様書の想定と異なる）
- 仕様書の指示同士が矛盾している、または既存の仕様書・`/docs/decisions.md` と衝突する
- 「設計の不変条件」（本ファイル後述）に違反する実装が要求されている
- Owner からの口頭依頼が `/specs/` 配下に対応する仕様書を持たない新機能である場合

## このプロジェクトの概要

Tryline は、海外ラグビーリーグ（Six Nations、Premiership、URC、Top 14、Rugby Championship、Super Rugby Pacific、RWC）を観戦する日本のラグビーファン向けの Web アプリです。試合データを集約し、LLM で日本語のプレビュー・レビューを生成し、試合ごとに AI チャットを提供します。

**ターゲットユーザー**: DAZN / J SPORTS / WOWOW に加入しており、週 3〜10 試合の国際試合を観戦し、日本語での分析コンテンツを求めている日本のラグビーファン。

**主要成功指標**: RWC 2027（2027年10〜11月）までの有料購読者数（¥980/月の Premium + 無料層）。

## 設計の不変条件

これらは交渉不可です。仕様書や Owner の依頼がこれらに違反している場合、Codex は実装を停止して矛盾を指摘してください。

- **試合中心のデータモデル**: すべてのコンテンツ（プレビュー、レビュー、チャットコンテキスト）は match_id に紐付く
- **LLM 生成コンテンツは試合単位でキャッシュ** し、ユーザー単位ではない。ユーザー数が増えてもコストは増えない（AI チャットを除く）
- **スクレイプした生テキストは決して再配信しない**。LLM で日本語に書き直してから配信する
- **robots.txt は常に尊重**。parser を使い、レート制限を守り、積極的にキャッシュする
- **コンテンツ生成パイプラインは 5 段階**: 集約 → 事実抽出 → Reddit フィルタ → ナラティブ生成 → 品質チェック（`specs/p1-content-pipeline.md` 参照）
- **著作権への配慮**: 15 語を超える直接引用なし、同一ソースから複数引用なし、原則として言い換える
- **モバイルファーストの PWA**、MVP ではネイティブアプリなし

## 技術スタック

- Next.js 15（App Router、RSC）
- TypeScript strict モード
- Supabase（postgres + auth + RLS）
- OpenAI API（抽出・QA は `gpt-4o-mini`、ナラティブは `gpt-4o`。モデル ID は `lib/llm/models.ts` で集中管理）
- Stripe（Phase 2 から決済）
- Vercel（ホスティング + cron）
- Tailwind CSS + shadcn/ui

## 本リポジトリでの作業方法

### Owner から実装依頼が来たとき

1. 指定された `/specs/<名前>.md` と `/docs/codex-prompts/<名前>.md` を必ず最初に読む
2. 依存する基盤仕様書（例: `p0-foundation.md`、`p1-data-model.md`、`p1-scraping-infra.md`）も確認する
3. `/docs/decisions.md` と `/docs/architecture.md` を読み、過去の判断と整合させる
4. 仕様書の「実装前のアクション」があれば、実装前に実施する
5. 仕様書の「範囲の再確認」と「対象外」を厳守する
6. 仕様書の「受け入れ条件」を全て満たすまで実装を続ける
7. 疑義があれば推測せず Owner に質問する

### PR を作成するとき

- ブランチ名は `codex/<仕様書スラッグ>` または Owner が指定した名前
- PR 説明に以下を含める:
  - 追加・変更したファイルの一覧と 1 行要約
  - 追加した依存パッケージとバージョン
  - 仕様書「完了時に必ず報告すること」の項目
  - 仕様書からの意図的な逸脱があれば、逸脱内容と理由
  - `pnpm lint` / `pnpm typecheck` / `pnpm test` の実行結果
- マージは Owner が行う。Codex は push / PR 作成までで停止する

### 仕様書と食い違う実装が必要と判明したとき

1. 実装を停止する
2. 食い違いの内容と、推奨する仕様書修正案を Owner に報告する
3. Claude Code が仕様書を改訂し、Owner がマージしたあとに実装を再開する
4. Codex 自身が仕様書（`specs/*.md`）や `CLAUDE.md` / `AGENTS.md` / `docs/decisions.md` を書き換えることは禁止。修正提案のみ

### デバッグ・障害調査を依頼されたとき

1. 再現手順と環境情報を Owner に確認する
2. 仕様書が想定していないケースなら、仕様書の不備として Owner に報告する
3. 修正を入れる場合は、まず失敗するテストを書いてから修正する

## Codex がやってはいけないこと

- 仕様書（`/specs/*.md`）の書き換え
- `CLAUDE.md` / `AGENTS.md` / `/docs/decisions.md` / `/docs/architecture.md` の書き換え（矛盾を見つけたら Owner に報告）
- 仕様書にない機能の追加・抽象化の前倒し
- Supabase の production プロジェクトに対する DDL / 書き込み
- RLS ポリシーの追加・変更（仕様書で明示されている場合を除く）
- 本番環境へのデプロイ
- プロダクトの方向性・優先順位の決定（Owner の判断）
- 新しい依存パッケージの追加（仕様書に明記されていない限り）
- `git push --force` / `git reset --hard` / ブランチ削除（Owner 承認必須）

## 仕様書が参照する構造（リファレンス）

Codex が読む `/specs/<名前>.md` は以下の構造を持ちます。Codex はこれらのセクションを読んで実装してください。

- **背景**: なぜ必要か、どのユーザー課題を解決するか
- **スコープ**: 対象 / 対象外
- **データモデル変更**: 追加・変更するテーブル、必要なマイグレーション
- **API サーフェス**: 新規ルート、型、リクエスト/レスポンス形式
- **UI サーフェス**: 画面、コンポーネント、状態遷移
- **LLM 連携**: パイプラインのどの段階か、どのモデルを使うか、プロンプトテンプレートへの参照
- **受け入れ条件**: 具体的で検証可能な条件。Codex は全項目を満たす
- **未解決の質問**: 実装前に Owner に確認する

## ドメイン用語集

Match = 2 チーム間の 1 試合
Competition = リーグまたはトーナメント（Six Nations、Premiership など）
Breakdown = ラック / モール周辺の接点戦術ゾーン（UI 用語ではない）
Lineout = ラインアウト（セットピース）
MOM = Man / Player of the Match（最優秀選手）
H2H = head-to-head、過去対戦成績
PWA = progressive web app（本プロダクトの配信形式）

## アンチパターン

以下のパターンを実装しそうになったら立ち止まり、仕様書と照合してください。

- 仕様書にない React コンポーネント・ページ・API ルートを追加する
- Match データモデルを経由しないコンテンツ保存（ユーザー ID 紐付けのキャッシュ等）
- LLM レスポンスを試合単位でなくユーザー単位でキャッシュする
- robots.txt を確認せずにスクレイプする、`lib/scrapers/fetchWithPolicy` を迂回する
- MVP の機能膨張 — Phase 1 仕様書にない機能を足す
- 汎用抽象化・プラグイン機構・設定システムの先回り実装（`Three similar lines is better than a premature abstraction`）
- エラーを握り潰して Promise を resolve する（throw が基本）
- バックワード互換性のためだけの死にコード（`_unused` 変数・`// removed` コメント等）を残す
